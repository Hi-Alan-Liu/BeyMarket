# BeyMarket 戰鬥陀螺零件交易賣場 — 專案企劃書

> 版本：v0.1 草案　|　日期：2026-07-07
> 定位：玩家對玩家(C2C)的戰鬥陀螺零件交易與行情平台（不含金流／發票）

---

## 1. 專案概述

### 1.1 一句話定位
一個讓戰鬥陀螺玩家**上架零件、搜尋零件、談成交易並記錄市場行情**的社群交易平台。平台本身**不經手金流與發票**，交易與付款透過玩家私下（主要為 LINE）完成，平台負責「媒合 + 資訊透明 + 行情紀錄」。

### 1.2 核心價值
- **上架方便**：賣家能快速把手上的零件依系列／類別分類上架。
- **搜尋方便**：買家能用系列、零件類型、關鍵字、價格區間快速找到想要的零件。
- **成交透明**：成交後由賣家手動勾選「已售出」並指定買家，形成交易紀錄。
- **行情透明**：以成交紀錄為基礎，統計每個零件「每日／每週」的成交價波動區間。
- **半實名**：僅開放 Discord / LINE 登入，降低詐騙與洗帳號的門檻。

### 1.3 明確不做（Out of Scope，第一版）
- ❌ 不串接金流（信用卡、第三方支付）
- ❌ 不開發票、不做金流對帳
- ❌ 不做站內即時聊天（溝通導流到 LINE）
- ❌ 不做物流／寄送整合
- ❌ 不做平台抽成拆帳

## 2. 目標使用者與情境

| 角色 | 目標 | 主要動作 |
|------|------|----------|
| 賣家 | 出清／販售手上零件 | 註冊登入 → 上架零件 → 收到詢問(LINE) → 手動標記已售出並指定買家 |
| 買家 | 找到特定零件或組裝需求 | 搜尋／篩選 → 查看行情 → 送出購買需求 or 直接加 LINE 洽談 |
| 觀望者 | 了解某零件行情 | 查看行情走勢頁，判斷合理價位 |

**核心使用者流程**
1. 買家搜尋零件 → 看到商品 + 近期行情。
2. 買家送出「購買需求」或點擊賣家 LINE 聯絡。
3. 雙方在 LINE 談定價格與交易方式。
4. 賣家回到平台，把該商品標記「已售出」→ 從詢問清單中選擇買家。
5. 系統將該筆成交價寫入行情資料，更新該零件的每日／每週波動。

## 3. 領域模型（戰鬥陀螺零件分類）

這是本專案最關鍵、也最需要正確設計的部分。零件分類必須「可擴充」，因為未來一定會出新系列與新零件類型。

### 3.1 三大系列（Series）
- **CX 系列**
- **UX 系列**
- **BX 系列**

### 3.2 零件主分類（Part Type）
一顆陀螺的組成 + 配件：

| 分類 | 說明 |
|------|------|
| 上蓋（Blade / Top） | 陀螺主體上蓋 |
| 固鎖（Ratchet / Lock） | 中間固定結構 |
| 軸心（Bit / Driver） | 底部軸心 |
| 發射器（Launcher） | 配件 |
| 拉條（Ripcord / Winder） | 配件 |
| 握把（Grip / Handle） | 配件 |
| 其他配件 | 保留彈性 |

### 3.3 CX 專屬擴充子分類（Sub-Type，僅 CX 有）
CX 系列的「上蓋／擴充結構」再細分：
- 紋章（Emblem）
- 戰刃（Main Blade）
- 輔助戰刃（Assist Blade）

> 設計重點：**系列(Series)**、**主分類(PartType)**、**子分類(SubType)** 三者採「可設定的分類表」而非寫死 enum，讓管理者未來能新增「DX 系列」或新零件類型而不需改程式。CX 的三種擴充子分類，透過「主分類 ↔ 允許的子分類」關聯表控制其只在 CX 出現。

### 3.4 商品（Listing）欄位建議
- 標題、描述
- 系列 / 主分類 / 子分類（CX 適用）
- 品項名稱（例：`DranBuster 3-60F`，可做正規化）
- 狀況（全新 / 二手 / 有使用痕跡）
- 開價（僅供參考，實際以 LINE 談定為準）
- 圖片（多張）
- 數量
- 賣家聯絡方式（LINE ID / 導流連結）
- 狀態（上架中 / 洽談中 / 已售出 / 下架）

## 4. 功能規格（第一版 MVP）

### 4.1 帳號與登入（半實名）
- **Discord OAuth 登入**
- **LINE Login（OAuth）**
- 首次登入建立平台個人檔案（暱稱、頭像、LINE 聯絡方式）。
- 不做 Email/密碼註冊，降低假帳號。

### 4.2 零件上架
- 選擇 系列 → 主分類 →（CX 才出現子分類）→ 填品項、狀況、開價、上傳圖片。
- 品項名稱可從「零件字典（Catalog）」自動建議，方便行情彙整。

### 4.3 列表與搜尋
- 全站列表 + 篩選器：系列、主分類、子分類、狀況、價格區間、關鍵字。
- 排序：最新、價格高低、熱門。
- 分頁 / 無限捲動。

### 4.4 購買需求
- 買家於商品頁「送出購買需求」（含留言、期望數量）。
- 賣家在後台看到該商品的所有需求者清單。
- 需求即通知（第一版可先 email/站內通知，之後接 LINE Notify）。

### 4.5 成交與已售出
- 賣家手動點「標記已售出」。
- 從「送出過需求的買家」中選擇實際成交對象（也允許手動輸入非站內買家）。
- 填入實際成交價 → 寫入行情資料。

### 4.6 市場波動紀錄（行情）
- 每筆成交寫入 `price_history`。
- 依「品項（正規化後的零件）」彙整：
  - 每日：最低 / 最高 / 平均 / 成交筆數
  - 每週：同上
- 商品頁與「行情頁」以折線圖 + 區間顯示波動。
- **全新 / 二手分開統計**（已定案）：因價差大，行情彙整時以「品項 × 狀況」為維度分別計算 min/max/avg，圖表可切換或並列顯示。
- 冷啟動策略：成交量少時，顯示「近 N 筆成交」而非強行畫日線。

### 4.7 賣家評價 / 信任分數（第一版納入）
> 依決策，第一版即納入，以降低無金流下的交易風險。
- 成交後買賣雙方可互相評價（星等 + 短評）。
- 賣家個人頁顯示：累積成交數、平均星等、信任分數。
- 信任分數初版公式：以成交筆數、平均評價、帳號註冊時長加權（可調）。
- 商品列表可顯示賣家星等徽章，協助買家判斷。
- 防刷機制：僅「已標記成交且指定為該買家」的交易可留評價，一筆交易一次評價。

## 5. 技術架構評估與建議

### 5.1 需求特性分析
- 流量規模：社群型、可預期為中小型（數百～數千活躍玩家）。
- 讀多寫少：大量搜尋瀏覽、少量上架與成交。
- 無金流 → **不需要 PCI 合規、不需要複雜交易一致性**，大幅降低複雜度。
- 需要 OAuth（Discord + LINE）。
- 需要圖片儲存。
- 需要行情彙整（定時或即時聚合）。

### 5.2 架構方案比較

| 方案 | 說明 | 優點 | 缺點 | 適合度 |
|------|------|------|------|--------|
| **A. Next.js 全端（推薦）** | Next.js(App Router) 前後端一體 + PostgreSQL | 一套語言(TS)、開發快、部署簡單、SEO 佳 | 需熟 React | ⭐⭐⭐⭐⭐ |
| B. 前後端分離 | React SPA + 獨立 Node/NestJS API | 職責清楚 | 兩套專案、初期過度設計 | ⭐⭐⭐ |
| C. 傳統框架 | Laravel / Django + Blade/Template | 後台成熟、開發快 | 前端互動較弱、行情圖較卡 | ⭐⭐⭐ |
| D. No-code | Bubble / Airtable | 極快上線 | 分類彈性、行情運算、OAuth 受限 | ⭐⭐ |

### 5.3 推薦技術棧（方案 A）

**核心**
- **框架**：Next.js 15（App Router）+ TypeScript — 前後端一體，API Routes / Server Actions 直接處理後端邏輯。
- **UI**：Tailwind CSS + shadcn/ui（快速做出乾淨列表與篩選介面）。
- **資料庫**：PostgreSQL（推薦 **Supabase** 或 **Neon** 託管）。
  - 關聯式非常適合「系列／分類／子分類」與「行情彙整」。
- **ORM**：Prisma（型別安全、Migration 方便）。
- **登入**：Auth.js（NextAuth v5）— 內建 Discord Provider，LINE 用 OIDC/自訂 Provider。
- **圖片儲存**：Supabase Storage 或 Cloudflare R2（便宜、免流量費）。
- **部署**：**Zeabur**（已定案）— 台灣在地、對 LINE 生態友善、支援自訂網域與 Cron。

**搜尋**
- 第一版：PostgreSQL 篩選 + `ILIKE` / 全文檢索（`tsvector`）即足夠。
- 未來量大再導入 **Meilisearch**（中文分詞友善、輕量）。

**行情彙整**
- 成交寫入 `price_history` 明細表。
- 每日用 **Cron（Vercel Cron / Supabase pg_cron）** 將明細彙整進 `price_daily_summary`。
- 前端用 Recharts / visx 畫波動折線與區間。

### 5.4 資料表初步規劃（PostgreSQL）

```
users            會員（Discord/LINE 綁定、LINE 聯絡方式）
series           系列（CX / UX / BX，可擴充）
part_types       主分類（上蓋/固鎖/軸心/發射器/拉條/握把…）
sub_types        子分類（紋章/戰刃/輔助戰刃）
part_catalog     零件字典（正規化品項，用於行情彙整）
listings         上架商品（賣家、分類、開價、狀況、狀態）
listing_images   商品圖片
purchase_requests 購買需求（買家、商品、留言）
transactions     成交紀錄（商品、賣家、買家、成交價、時間）
price_history    行情明細（由 transactions 產生，含 condition 狀況欄位）
price_daily_summary  每日彙整（依 品項 × 狀況 分組：min/max/avg/count）
price_weekly_summary 每週彙整（依 品項 × 狀況 分組）
reviews          評價（交易、評價者、被評者、星等、短評）
seller_stats     賣家統計（成交數、平均星等、信任分數，可由 reviews 彙整）
```

分類彈性關鍵：`part_types` ↔ `sub_types` 用關聯控制「哪個主分類允許哪些子分類」（只有 CX 相關主分類才掛紋章/戰刃/輔助戰刃）。

### 5.5 為什麼推薦這套
1. **不需金流 → 選最省事的全端方案**，把力氣放在分類設計與行情。
2. **TypeScript 一條龍**，一個人／小團隊也能維護。
3. **PostgreSQL 關聯式** 完美吻合分類階層與行情聚合需求。
4. **Auth.js** 直接支援 Discord，LINE 也能透過 OIDC 整合，符合半實名。
5. **託管服務（Supabase + Vercel/Zeabur）** 幾乎零維運，成本低。

### 5.6 成本概估（起步階段）
- Supabase 免費層 / Vercel 免費層即可跑 MVP。
- 圖片流量成長後：Cloudflare R2（幾乎無流量費）。
- 初期月成本可壓在 **NT$0～500** 區間。

## 7. 風險與注意事項
- **詐騙風險**：無金流保障，需在站上明確標示「平台不介入交易糾紛」，並靠半實名 + 賣家評價（第一版已納入）降低風險。
- **行情冷啟動**：成交量不足時波動線沒意義，需以「近 N 筆」呈現。
- **LINE Login 審核**：LINE Login channel 需申請與網域驗證，需預留時間。
- **零件命名混亂**：務必用「零件字典(Catalog)」正規化品項，否則行情無法彙整。
